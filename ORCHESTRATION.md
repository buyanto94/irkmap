Backend, база данных, серверная авторизация, Sphinx, хранение файлов и email-рассылки не ставятся исполнителю как задачи в этом репозитории. Если фронтенду нужен backend-контракт, фиксируем только ожидаемые API/данные и временные mock-решения (см. `FRONTEND_API_CONTRACT.md`).

## Как работать

0. **ВАЖНО:** Исполнитель не редактирует этот файл без отдельного задания. Статусы меняет только оркестратор.
1. Исполнитель берёт только один блок из раздела «Задания для исполнителя».
2. Перед кодингом исполнитель читает файлы, указанные в задании, и проверяет существующие паттерны (FSD-структура `src/`, Composition API + `<script setup>`, Pinia, Tailwind).
3. Исполнитель меняет только файлы, относящиеся к заданию.
4. После выполнения исполнитель запускает проверки из задания.
5. Оркестратор проверяет результат, переносит задание в «Историю», добавляет замечания или следующее задание.

## Команды проверки

Базовая проверка (типы + сборка):

```bash
npm install   # один раз, если нет node_modules
npm run build
```

Если менялись Vue/TS/CSS файлы:

```bash
npm run lint
```

Тестов в проекте пока нет. Когда появятся — добавить сюда команду запуска.

## Архитектура (как есть)

- Vue 3 + Vite + TypeScript, Composition API, `<script setup>`.
- Структура по Feature-Sliced Design: `src/app` (точка входа, router, stores, css), `src/pages`, `src/entities`, `src/shared` (ui, api).
- Pinia store: `src/app/stores/tourism.ts` — единый источник данных по объектам и категориям.
- Доменные типы: `src/entities/tourism/model/types.ts` (роли, статусы, объект, заявка, лог и т.д.).
- HTTP: `src/shared/api/axiosClient.ts`, базовый URL через `VITE_API_BASE_URL`.
- Mock API: статические JSON в `public/api/` (`categories.json`, `objects.json`).
- Карта: `src/shared/ui/InteractiveMap.vue` — Leaflet + OSM + markercluster.
- UI-библиотеки: Tailwind 4, HeadlessUI, Lucide, Swiper, Fancybox, vue-easy-lightbox.

`npm install` + `vue-tsc --build` проходят без ошибок. Реализована публичная часть на уровне, близком к MVP.

- Доменные типы фронтенда: роли (`ADMIN/MODERATOR/BUSINESS/USER`), статусы пользователя и объекта, заявка бизнеса, модерационный комментарий, лог.

### Известные расхождения с ТЗ (бэклог)

Публичная часть:

1. **Open Graph статичный и фейковый.** В `index.html` зашиты `example.com`; нет per-page мета-тегов и `og:title/og:image` под конкретный объект.
2. **Карта на детальной — заглушка.** Блок «Статичная карта» и кнопка «Построить маршрут» ничего не делают.
3. **Состояния загрузки/ошибки/пустого результата** показаны только на детальной; на главной и в каталоге их нет.
4. **Фильтр расстояния** работает по статическому полю `distance`, без геолокации пользователя.
5. **«Показать телефон»** использует `alert()`; нет учёта кликов (аналитика), как просит ТЗ.
6. **Брендинг** — в шапке заглушка «Logo».
7. **Технический долг (lint).** Пре-существующие ошибки `@typescript-eslint/no-explicit-any` в `Catalog.vue` (строки ~27/77/117) и неиспользуемый `ChevronLeft` в `ObjectDetails.vue`; плюс новые `any` в `goToObject` (Catalog/Home) от TASK-101. `npm run lint` сейчас не зелёный. Дублируется логика вычисления `categorySlug` (find by name → slug, fallback `misc`) в 4 местах. → см. TASK-101F.
8. **`handleMarkerClick`** в каталоге всё ещё только пишет в консоль (не критично — навигация идёт через попап).

Личные кабинеты (в ТЗ есть, во фронтенде отсутствуют полностью):

9. **Авторизация/регистрация** — кнопки «Войти»/«Регистрация» неактивны; нет экранов и mock-стора авторизации.
10. **Бизнес-кабинет** — нет списка своих объектов со статусами, формы создания/редактирования (категория, rich-text описание, drag-and-drop фото + выбор главного, маркер на карте), кнопки «Опубликовать».
11. **Двухэтапная регистрация бизнеса** — нет формы заявки (email, компания, ИНН, контактное лицо, загрузка до 5 документов).
12. **Кабинет модератора** — нет очереди модерации с фильтром по статусу, карточки проверки, действий «Опубликовать/Отклонить/На корректировку» (с обязательной причиной), Diff View старой/новой версии, очереди верификации бизнес-заявок.
13. **Кабинет администратора** — нет CRUD категорий (иконка, автогенерация slug, drag-and-drop сортировка), управления пользователями, журнала действий, дашборда статистики.

## Граница ответственности фронтенда

Фронтенд **реализует**: публичные страницы, маршрутизацию и ЧПУ, адаптивную вёрстку, работу с API через единый слой, состояния загрузки/ошибок/пустых результатов, формы и валидацию, UX личных кабинетов, UI для всех ролей, UI модерации/заявок/статусов/журналов, mock API или fixtures до готовности backend, описание ожидаемых backend-контрактов.

Фронтенд **не реализует** как финальную логику: хранение пользователей и ролей, серверную авторизацию и права доступа, серверную модерацию, загрузку и хранение файлов, отправку писем, Sphinx/полнотекстовый поиск. Допустимы временные frontend-mocks для демонстрации сценариев, если они явно помечены как mock.

## Приоритеты

- **P0 — стабилизация (готово).** Кодировка, единый store, рабочие фильтры, README, зелёная сборка. Закрыто.
- **P1 — допилить публичную часть до полного MVP.** ЧПУ, динамический Open Graph, карта/маршрут на детальной, SPA-навигация из попапа, состояния загрузки/ошибок/пустого на главной и в каталоге, аналитика кликов по контактам. → TASK-101…104.
- **P2 — фронтенд личных кабинетов на mock/контрактах.** Авторизация, бизнес-кабинет, модерация, админка. → TASK-201…205.
- **P3 — подготовка к реальному backend.** Заменить mock endpoints на реальные через `VITE_API_BASE_URL`, обработать ошибки API и права доступа, убрать временные mock-ветки. → TASK-301.

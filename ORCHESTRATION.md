# Оркестрация фронтенд-разработки IRK Map

Этот файл - рабочий документ для связки "Codex-оркестратор -> другая ИИ-исполнитель".

Важно: этот репозиторий содержит только фронтенд-часть проекта. Backend, база данных, серверная авторизация, Sphinx, хранение файлов и email-рассылки не должны ставиться исполнителю как задачи в этом репозитории. Если фронтенду нужен backend-контракт, здесь фиксируем только ожидаемые API/данные и временные mock-решения.

Codex не пишет основной код без отдельной просьбы. Его задача: читать ТЗ и текущую реализацию, находить расхождения во фронтенде, формулировать маленькие понятные задания, принимать результат, фиксировать что сделано и что нужно переделать.

## Как работать

1. Исполнитель берет только один блок из раздела "Задания для исполнителя".
2. Перед кодингом исполнитель читает файлы, указанные в задании, и проверяет существующие паттерны.
3. Исполнитель меняет только файлы, относящиеся к заданию.
4. После выполнения исполнитель запускает проверки из задания.
5. Оркестратор проверяет результат и обновляет этот файл: переносит задание в "Историю", добавляет замечания или следующее задание.

## Команды проверки

Базовая проверка:

```bash
npm run build
```

Дополнительно, если менялись Vue/TS/CSS файлы:

```bash
npm run lint
```

Если добавятся фронтенд-тесты, этот раздел нужно дополнить командой тестов.

## Краткий вывод по текущему состоянию

Проект сейчас является Vue 3 + Vite фронтендом с моковыми JSON в `public/api`.

Сейчас реализовано частично:

- публичная главная страница;
- публичный каталог;
- детальная страница объекта;
- Leaflet/OpenStreetMap карта;
- локальные фильтры по категории и тексту;
- галерея на детальной странице;
- моковые категории и объекты.

Критичные фронтенд-проблемы:

- В русских строках проекта испорчена кодировка: вместо нормального текста видны `Рџ`, `Рљ`, `СЃ` и похожие последовательности. Это есть в `TECH_SPEC.md`, Vue-файлах и моковых JSON.
- README остался от шаблона AI Studio/Gemini и не описывает проект.
- `src/shared/store/useObjectsStore.ts` выглядит как неиспользуемый старый моковый store и дублирует `src/app/stores/tourism.ts`.
- Маршруты ЧПУ из ТЗ не реализованы в нужном виде: сейчас объект открывается как `/objects/:slug`, а в ТЗ пример `/hotels/gostinica-uyut`.
- Карта без кластеризации и без карточек-попапов.
- Кнопка "Искать в этом районе" только логирует событие и не фильтрует по bounds.
- Фильтр по расстоянию и сортировка по рейтингу визуально есть, но не работают.
- Детальная страница содержит заглушку статической карты вместо карты/маршрута.
- Нет Open Graph metadata для публичных страниц.
- Нет фронтенд-экранов личных кабинетов, модерации и администрирования, хотя они описаны в ТЗ.

## Граница ответственности фронтенда

Фронтенд должен реализовать:

- публичные страницы;
- маршрутизацию и ЧПУ;
- адаптивную верстку;
- работу с API через единый слой;
- состояния загрузки, ошибок и пустых результатов;
- формы, валидацию и UX личных кабинетов;
- UI для ролей: пользователь, бизнес-пользователь, модератор, администратор;
- UI для модерации, заявок, карточек объектов, статусов и журналов;
- mock API или fixtures до готовности backend;
- описание ожидаемых backend-контрактов, если их нет.

Фронтенд не должен реализовывать как финальную логику:

- хранение пользователей и ролей;
- серверную авторизацию;
- серверную модерацию;
- загрузку и хранение файлов;
- отправку писем;
- Sphinx/полнотекстовый поиск на сервере;
- реальные права доступа.

Допустимо делать временные frontend mocks для демонстрации сценариев, если они явно помечены как mock.

## Как работать

0. **ВАЖНО:** Исполнитель не редактирует этот файл без отдельного задания. Статусы меняет только оркестратор.
1. Исполнитель берёт только один блок из раздела «Задания для исполнителя».
2. Перед кодингом исполнитель читает файлы, указанные в задании, и проверяет существующие паттерны (FSD-структура `src/`, Composition API + `<script setup>`, Pinia, Tailwind).
3. Исполнитель меняет только файлы, относящиеся к заданию.
4. После выполнения исполнитель запускает проверки из задания.
5. Оркестратор проверяет результат, переносит задание в «Историю», добавляет замечания или следующее задание.

## Архитектура (как есть)

- Vue 3 + Vite + TypeScript, Composition API, `<script setup>`.
- Структура по Feature-Sliced Design: `src/app` (точка входа, router, stores, css), `src/pages`, `src/entities`, `src/shared` (ui, api).
- Pinia store: `src/app/stores/tourism.ts` — единый источник данных по объектам и категориям.
- Доменные типы: `src/entities/tourism/model/types.ts` (роли, статусы, объект, заявка, лог и т.д.).
- HTTP: `src/shared/api/axiosClient.ts`, базовый URL через `VITE_API_BASE_URL`.
- Mock API: статические JSON в `public/api/` (`categories.json`, `objects.json`).
- Карта: `src/shared/ui/InteractiveMap.vue` — Leaflet + OSM + markercluster.
- UI-библиотеки: Tailwind 4, HeadlessUI, Lucide, Swiper, Fancybox, vue-easy-lightbox.

### Известные расхождения с ТЗ (бэклог)

Публичная часть:

1. **Фильтр расстояния** работает по статическому полю `distance`, без геолокации пользователя (Geolocation API).
2. **Роль-guard неполный.** `beforeEach` проверяет только `requiresAuth`, не роль. Пункты меню кабинетов — заглушки `to="#"`. Когда появятся реальные роуты модерации/админки (TASK-203/205), добавить проверку роли в `meta` + guard.
3. **Регистрация бизнеса срезает ТЗ-флоу.** `register-business` сразу логинит как `BUSINESS`/`ACTIVE`, минуя двухэтапный процесс (заявка → `PENDING` → подтверждение модератором → `ACTIVE`). Корректную постановку в очередь верификации доделать в TASK-204 (статус `PENDING`, без моментального входа).
4. **OG только client-side.** `useMeta` обновляет теги через JS в рантайме. Этого хватает для in-app навигации и краулеров, исполняющих JS, но классические соц-скраперы (VK/Telegram/Facebook) JS не исполняют и увидят дефолтные теги из `index.html`, а не per-object. Полноценный per-page OG для шеринга требует SSR/пререндера — вне рамок фронтенд-репо, зафиксировано как ограничение для backend/инфраструктуры.
4. **`handleMarkerClick`** в каталоге всё ещё только пишет в консоль (не критично — навигация идёт через попап).
5. **Аналитика и маршрут на mock.** Клики логируются в `console.log` (`[MOCK ANALYTICS]`), кнопка телефона после раскрытия не является `tel:`-ссылкой, маршрут открывает Яндекс.Карты (ТЗ допускает «в крайнем случае Яндекс»). Финализируется при интеграции backend.

Личные кабинеты (в ТЗ есть, во фронтенде отсутствуют полностью):

9. **Авторизация/регистрация** — кнопки «Войти»/«Регистрация» неактивны; нет экранов и mock-стора авторизации.
10. **Бизнес-кабинет** — нет списка своих объектов со статусами, формы создания/редактирования (категория, rich-text описание, drag-and-drop фото + выбор главного, маркер на карте), кнопки «Опубликовать».
11. **Двухэтапная регистрация бизнеса** — нет формы заявки (email, компания, ИНН, контактное лицо, загрузка до 5 документов).
12. **Кабинет модератора** — нет очереди модерации с фильтром по статусу, карточки проверки, действий «Опубликовать/Отклонить/На корректировку» (с обязательной причиной), Diff View старой/новой версии, очереди верификации бизнес-заявок.
13. **Кабинет администратора** — нет CRUD категорий (иконка, автогенерация slug, drag-and-drop сортировка), управления пользователями, журнала действий, дашборда статистики.

## Граница ответственности фронтенда

Фронтенд **реализует**: публичные страницы, маршрутизацию и ЧПУ, адаптивную вёрстку, работу с API через единый слой, состояния загрузки/ошибок/пустых результатов, формы и валидацию, UX личных кабинетов, UI для всех ролей, UI модерации/заявок/статусов/журналов, mock API или fixtures до готовности backend, описание ожидаемых backend-контрактов.

Фронтенд **не реализует** как финальную логику: хранение пользователей и ролей, серверную авторизацию и права доступа, серверную модерацию, загрузку и хранение файлов, отправку писем, Sphinx/полнотекстовый поиск. Допустимы временные frontend-mocks для демонстрации сценариев, если они явно помечены как mock.

### TASK-105F2: Высота корневого контейнера — `min-h-screen` → `min-h-dvh` в App.vue

Статус: **готово к выдаче (следующая по очереди). Причина подтверждена: фикс применён оркестратором локально, владелец проверил на реальном телефоне — пустой скролл ушёл; правка откачена для передачи через исполнителя.**

Подтверждённая причина (важно — ровно одна строка): корневой контейнер приложения в `src/app/App.vue` использует `min-h-screen` (= `min-height: 100vh`). На мобильных `100vh` — это «большой» вьюпорт, БЕЗ учёта адресной строки браузера. Когда адресная строка видна, контейнер выше видимой области ровно на её высоту → появляется внешняя вертикальная прокрутка с пустотой снизу. При этом layout'ы (`DefaultLayout`/`DashboardLayout`) уже используют `h-dvh` (динамическая высота), поэтому конфликт `100vh` (App.vue) ↔ `dvh` (layout) и даёт пустой отступ под адресной строкой. Подсказка владельца «из-за блока где урл сайта» = именно адресная строка.

Что сделать (1 файл, 1 строка):

1. В `src/app/App.vue` заменить класс корневого `<div>` `min-h-screen` → `min-h-dvh`.

Если на каком-то устройстве остаточный скролл сохранится — запасной вариант `min-h-svh` (small viewport height, гарантированно не больше видимой области). Но проверенный и достаточный вариант — `min-h-dvh`.

Не нужно: трогать `DefaultLayout`, добавлять проп `fullscreen`, менять отступы каталога/шторки или drag-логику. `pb-4` из TASK-105F можно оставить — не мешает.

Критерии приёмки: на мобильном (реальное устройство/эмуляция с адресной строкой) при поднятой шторке нет внешней прокрутки и пустой полосы снизу; десктоп и остальные страницы без изменений; `npm run build` и `npm run lint` проходят.
